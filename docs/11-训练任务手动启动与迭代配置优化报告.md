# 训练任务手动启动与迭代配置优化报告

## 1. 背景

此前 App 点击“创建重建任务”后，如果后端以 `BACKEND_AUTO_START_PIPELINE=true` 启动，任务会在上传完成后直接进入训练流水线。这个行为不适合后续引入训练设置，因为用户没有机会在训练前选择质量档位、训练轮数或未来的 object masking 等选项。

另外，`splatfacto` 默认 `30000` 轮对本地 MVP 来说等待时间偏长。当前更合理的默认策略是先用 `7000` 轮快速验证质量，再按需切换到 `30000` 轮做高质量对比。

## 2. 本次目标

本次改动目标：

- 创建任务只负责上传视频并生成 `task_id`，不再自动启动训练。
- 增加手动启动流水线接口。
- 增加终止流水线接口，便于中途放弃高成本任务。
- 将质量档位、训练轮数和 object masking 预留开关纳入任务配置。
- 将默认训练轮数从 `30000` 改为 `7000`。
- 在 Flutter 端增加“训练设置”区域，用户确认配置后再启动流水线。

## 3. 后端接口变化

### 3.1 创建任务

接口仍然是：

```text
POST /api/v1/reconstructions
```

现在它只做：

- 保存上传视频。
- 创建 task JSON。
- 返回状态为 `uploaded` 的任务。

它不再触发 `trainer.pipeline` 子进程。

### 3.2 手动启动训练

新增接口：

```text
POST /api/v1/reconstructions/{task_id}/pipeline/start
```

请求体：

```json
{
  "quality_profile": "balanced",
  "train_max_steps": 7000,
  "object_masking": false
}
```

字段说明：

- `quality_profile`：训练质量档位，当前支持 `fast / balanced / quality / raw`。
- `train_max_steps`：训练最大迭代步数，当前只允许 `7000` 或 `30000`。
- `object_masking`：预留字段，当前只记录配置，不改变训练行为。
- `mock_mode`：可选字段；未传时仍沿用 `BACKEND_PIPELINE_MOCK`，便于本地联调。

接口限制：

- 只有 `uploaded` 或 `failed` 状态可以启动。
- 其他状态会返回 `409`，避免重复启动正在处理的任务。
- 启动前会校验视频文件仍存在。
- 配置确认后会先把任务置为 `queued`，防止子进程启动到写入 `preprocessing` 之间被重复点击。

### 3.3 终止训练

新增接口：

```text
POST /api/v1/reconstructions/{task_id}/pipeline/cancel
```

接口行为：

- 只允许 `queued / preprocessing / training / exporting` 状态终止。
- 后端优先使用 task JSON 中记录的 `pipeline_pid` 终止进程树。
- 如果老任务没有 `pipeline_pid`，会在 Windows 上按 `task_id` 从进程命令行里查找并终止相关进程。
- 成功后任务状态改为 `cancelled`，保留已有日志和进度，后续可以重新进入训练设置页再次启动。

## 4. Pipeline 变化

### 4.1 默认训练轮数

新增默认值：

```text
DEFAULT_TRAIN_MAX_STEPS = 7000
```

允许选项：

```text
TRAIN_MAX_STEP_OPTIONS = 7000, 30000
```

`trainer.pipeline` 新增参数：

```powershell
python -m trainer.pipeline `
  --task-id <task_id> `
  --input-video <video> `
  --quality-profile balanced `
  --train-max-steps 7000
```

### 4.2 ns-train 命令

训练命令现在显式传入：

```text
--max-num-iterations <train_max_steps>
```

因此 `7000` 轮任务会按 `7000` 作为训练总进度，`30000` 轮任务会按 `30000` 作为训练总进度。

### 4.3 任务状态记录

task JSON 新增/使用以下字段：

```json
{
  "quality_profile": "balanced",
  "train_max_steps": 7000,
  "object_masking": false,
  "pipeline_pid": 12345,
  "train_step": 0,
  "train_total_steps": 7000
}
```

这些字段会通过任务详情接口返回给 Flutter。

## 5. Flutter 端变化

### 5.1 上传后的流程

旧流程：

```text
点击创建重建任务 -> 上传 -> 后端直接开始流水线 -> 进入状态页
```

新流程：

```text
点击创建重建任务 -> 上传 -> 进入任务状态页 -> 显示训练设置 -> 用户点击开始训练流水线
```

### 5.2 训练设置卡片

当任务状态为 `uploaded / failed / cancelled` 时，`TaskStatusPage` 会显示“训练设置”卡片：

- 质量档位下拉选择：`Fast / Balanced / Quality / Raw`
- 训练轮数选择：`7000 轮 / 30000 轮`
- Object masking 预留开关：当前禁用并说明后续接入
- “开始训练流水线”按钮

点击按钮后，Flutter 调用：

```text
POST /api/v1/reconstructions/{task_id}/pipeline/start
```

然后页面继续轮询任务状态。

当任务状态为 `queued / preprocessing / training / exporting` 时，`TaskStatusPage` 会显示“流水线控制”卡片，可手动点击“终止流水线”。

## 6. 预期影响

- 用户可以在训练前选择质量/耗时取舍。
- 默认 `7000` 轮会明显缩短 MVP 验证时间。
- 仍保留 `30000` 轮用于高质量对比和最终产物验证。
- 后续接入 object masking 时，不需要再重构任务配置入口，只需要让预留字段真正参与预处理链路。

## 7. 后续建议

建议下一步验证：

1. 创建新任务后确认状态停留在 `uploaded`，不会自动训练。
2. 在 App 的训练设置卡片选择 `balanced + 7000`，点击开始训练。
3. 点击后确认任务先进入 `queued`，随后进入 `preprocessing / training`。
4. 检查 `pipeline.log` 中 `ns-train` 命令包含 `--max-num-iterations 7000`。
5. 对比 `7000` 与 `30000` 的训练耗时、Gaussian 数量、PLY 大小和 Viewer 主观效果。
6. 若 `7000` 质量不足，再决定是否把默认值调到中间档，例如未来新增 `12000` 或 `15000`。
