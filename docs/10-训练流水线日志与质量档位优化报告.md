# 训练流水线日志与质量档位优化报告

## 1. 背景

本次优化源于一次模型质量分析：已完成任务 `task_20260406_230817_4553a8` 的训练轮数为 `30000`，但 Viewer 中细节偏低。排查发现主要问题不在 PlayCanvas Viewer，而在训练输入侧：

- 源视频为 `3840x2160`，带 `rotation=-90` 旋转元数据。
- 旧转码逻辑按 `ffprobe` 原始 `width/height` 判断缩放方向，实际经 FFmpeg 自动旋转后输出成了 `1920x3414`。
- nerfstudio 训练时又自动选择 `Auto image downscale factor of 4`，实际训练图像接近 `480x853`。
- 这会明显限制商品纹理和几何细节，继续单纯增加训练轮数收益有限。

## 2. 本次目标

本次改动覆盖三件事：

- 建立每个 task 独立的 pipeline 日志系统，记录完整阶段、命令、耗时和关键参数。
- 引入训练质量档位，支持按 `fast / balanced / quality / raw` 控制输入分辨率和 nerfstudio downscale。
- 修正 FFmpeg 缩放逻辑，使带旋转元数据的视频也能按实际输出画面的最长边限制分辨率。

## 3. 日志系统改动

每个任务仍然使用：

```text
storage/processed/<task_id>/pipeline.log
```

日志格式改为带时间信息的统一格式：

```text
2026-04-08T00:06:15+08:00 +0.10s [INFO] [normalize] Video normalization configured | {...}
```

字段含义：

- 第一段是本地时间戳，包含时区。
- `+0.10s` 是从当前 pipeline task 开始到该日志行的累计耗时。
- `[INFO] / [STATUS] / [CMD] / [WARN] / [ERROR]` 是日志级别。
- `[normalize] / [probe] / [ns-train]` 等是 pipeline 阶段。
- `| {...}` 后面是结构化 JSON 字段，记录命令、路径、分辨率、profile、returncode、duration 等。

现在会记录的关键内容：

- task 启动参数：`task_id`、`input_video`、`mock`、`quality_profile`、Python 路径。
- `ffprobe` 开始/结束、耗时、源视频元数据。
- `ffmpeg` 转码配置、命令、耗时、转码后分辨率。
- 真实训练链路中的 `ffmpeg / ffprobe / colmap / ns-process-data / ns-train / ns-export` 可执行文件检查。
- `ns-process-data`、`ns-train`、`ns-export` 的开始/结束、returncode、耗时和关键 stdout 行。
- `colmap_compat.py` 执行 COLMAP 子命令时，会把最终命令行和 returncode 写入 `pipeline.log`，并同步写到任务目录下的 `colmap_compat.log`。
- COLMAP sparse model 分析结果，以及最终选择的最佳模型。
- mock 或真实模型导出路径、模型文件大小。
- 失败时记录 `Pipeline task failed`，并把错误同步到 task JSON。

## 4. 质量档位

新增 `--quality-profile` 参数，默认值来自环境变量 `TRAINING_QUALITY_PROFILE`，没有设置时为 `balanced`。

```powershell
python -m trainer.pipeline --task-id <task_id> --input-video <video> --quality-profile balanced
```

档位定义如下：

| 档位 | FFmpeg 最长边 | `ns-process-data --num-downscales` | 训练 `--downscale-factor` | `pipeline.model.num-downscales` | 用途 |
|---|---:|---:|---:|---:|---|
| `fast` | 1600 | 2 | 2 | 2 | 快速验证，成本最低 |
| `balanced` | 2560 | 2 | 2 | 2 | 默认档，兼顾细节和训练成本 |
| `quality` | 3200 | 2 | 2 | 1 | 更高质量，要求更好的素材和显存 |
| `raw` | 不主动缩放 | 2 | 1 | 1 | 实验档，容易显存/耗时失控 |

当前后端不再在创建任务后自动启动 pipeline。质量档位和训练轮数会在 App 训练设置页确认后传给 `pipeline/start` 接口；环境变量只作为默认值和独立 worker 的兜底配置：

```powershell
$env:TRAINING_QUALITY_PROFILE = "balanced"
$env:TRAINING_MAX_STEPS = "7000"
$env:BACKEND_PIPELINE_MOCK = "false"
```

独立 worker 也支持：

```powershell
python -m trainer.worker --quality-profile quality --train-max-steps 30000
```

## 5. FFmpeg 缩放修正

旧逻辑：

```text
如果 ffprobe width >= height，则 scale=1920:-2
否则 scale=-2:1920
```

这个逻辑在手机视频带旋转元数据时会出错。源文件 `3840x2160 + rotation=-90` 经 FFmpeg 自动旋转后实际是竖屏，旧逻辑仍然固定输出宽度，导致得到 `1920x3414`。

新逻辑使用 FFmpeg 在滤镜阶段的实际画面尺寸，并限制输出边界：

```text
fps=3,scale=min(<max_edge>\,iw):min(<max_edge>\,ih):force_original_aspect_ratio=decrease:force_divisible_by=2
```

效果：

- 使用 FFmpeg 的 post-autorotation 尺寸判断。
- 保持原始宽高比。
- 最长边不会超过档位上限。
- 不主动放大小于上限的视频。
- 输出宽高保持偶数，避免 H.264 编码问题。

## 6. 训练命令变化

旧训练命令核心部分：

```powershell
ns-train splatfacto --data <dataset> --vis tensorboard --pipeline.datamanager.cache-images cpu
```

新训练命令改为显式指定 nerfstudio-data dataparser：

```powershell
ns-train splatfacto `
  --max-num-iterations <train_max_steps> `
  --vis tensorboard `
  --logging.local-writer.max-log-size 0 `
  --logging.steps-per-log 10 `
  --pipeline.datamanager.cache-images cpu `
  --pipeline.model.num-downscales <profile.model_num_downscales> `
  nerfstudio-data `
  --data <dataset> `
  --downscale-factor <profile.train_downscale_factor>
```

这样可以避免 nerfstudio 自动按 `<1600px` 规则选择过强 downscale，例如之前的 `Auto image downscale factor of 4`。

后续已将默认 `train_max_steps` 改为 `7000`，并保留 `30000` 作为高质量对比选项；详见 `docs/11-训练任务手动启动与迭代配置优化报告.md`。

`ns-process-data video` 也增加了：

```powershell
--num-downscales <profile.process_num_downscales>
```

当前默认只生成 `images_2` 和 `images_4`，减少不必要的 `images_8` 产物。

## 7. COLMAP 兼容层

`ns-process-data` 仍然负责调用 COLMAP，但所有 COLMAP 子命令会经过 `trainer/colmap_compat.py`。

兼容层当前只做两件事：

- 把旧参数名 `--SiftExtraction.use_gpu` 改写为 `--FeatureExtraction.use_gpu`。
- 把旧参数名 `--SiftMatching.use_gpu` 改写为 `--FeatureMatching.use_gpu`。

兼容层不再主动追加 `--Mapper.ba_use_gpu` 或 `--BundleAdjustment.use_gpu`。原因是本地 COLMAP 虽然是 CUDA build，但它链接的 Ceres 没有 CUDA/cuDSS 支持；即使命令行请求 GPU BA，COLMAP 也会回退到 CPU solver。该问题已记录在 `docs/12-COLMAP位姿恢复与BA无法使用GPU遗留问题报告.md`。

为了避免后续只能通过任务管理器猜测 GPU 参数是否生效，`colmap_compat.py` 会在每次调用 COLMAP 时记录类似下面的诊断行：

```text
[colmap_compat] Executing ... colmap.exe mapper ...
[colmap_compat] Finished returncode=0
```

这些诊断行会进入外层 `pipeline.log`，也会写入当前任务处理目录下的 `colmap_compat.log`。

## 8. 验证结果

已完成的验证：

- `python -m py_compile trainer\pipeline.py trainer\worker.py backend\app\routes\reconstructions.py` 通过。
- `ns-train splatfacto ... nerfstudio-data --data ... --downscale-factor 2 --help` 可正常解析，说明命令结构符合 nerfstudio CLI。
- 用已有源视频跑 mock pipeline 验证日志和转码尺寸。

同一个 `3840x2160 + rotation=-90` 源视频的验证结果：

| 档位 | 转码后尺寸 |
|---|---|
| `balanced` | `1440x2560` |
| `fast` | `900x1600` |

这说明新 FFmpeg 缩放逻辑已经不再产生旧的 `1920x3414` 长边失控结果。

## 9. 预期影响

对训练质量：

- `balanced` 预计比旧结果更清晰，因为训练输入不再被自动落到 `480x853` 级别。
- `quality` 会进一步提高输入细节，但会增加 COLMAP、训练和导出压力。
- `raw` 只适合手动实验，不建议作为默认 App 任务档位。

对 Viewer：

- 本次没有修改 PlayCanvas Viewer 渲染逻辑。
- 之前的低清问题主要来自训练输入分辨率，不是 Viewer 主动压缩模型。
- Viewer 端后续仍可单独评估 DPR 上限、加载格式压缩和移动端性能。

## 10. 后续建议

下一步建议按以下顺序验证真实质量收益：

1. 用同一段商品视频分别跑 `balanced` 和旧模型对比。
2. 检查新日志里是否仍出现 `Auto image downscale factor`，以及 config 中 `downscale_factor` 是否为预期值。
3. 对比 `model.ply` 大小、导出 Gaussian 数量、Viewer 主观细节。
4. 如果 `balanced` 仍偏糊，再跑 `quality`。
5. 如果显存不足，先退回 `balanced`，再考虑商品裁剪、mask 或限制输入帧数，而不是直接使用 `raw`。
