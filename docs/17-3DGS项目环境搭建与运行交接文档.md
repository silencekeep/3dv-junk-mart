# 1. 项目概述

## 1.1 当前项目已经实现了什么

当前仓库已经具备一个可本地跑通的 3DGS 二手商品 MVP，包含以下核心能力：

1. Flutter App 基础框架
2. 上传商品视频并创建重建任务
3. FastAPI 后端任务管理与文件存储
4. 3DGS 重建流水线
5. Object Masking 交互（基于 SAM2）
6. 高斯导出、PLY 导出、SOG 导出
7. Web Viewer 查看与姿态/初始视角编辑
8. 商品发布流转，以及首页/详情页内嵌查看

## 1.2 整体架构

当前项目不是“前后端分离 + 独立训练服务 + 独立对象存储 + 独立数据库”的生产架构，而是一个**本机单仓库、本地文件存储、本地进程编排**的 MVP。

核心目录职责如下：

- `mobile_app/`：Flutter App，负责上传视频、任务管理、Masking 交互、Viewer 嵌入、发布页/详情页展示
- `backend/`：FastAPI 服务，对外提供任务、发布、Viewer 配置等 API
- `trainer/`：训练与导出流水线脚本，包括预处理、Nerfstudio、SAM2、对象提取、SOG 导出
- `viewer/`：静态 Web Viewer，由后端直接挂载
- `shared/`：前后端共享的文件存储和任务元数据逻辑
- `storage/`：本地数据目录，保存上传文件、处理中间产物、模型产物、任务 JSON

## 1.3 前后端与训练链路的关系

当前真实链路如下：

1. Flutter App 调用后端上传视频，后端在 `storage/uploads/` 下保存原视频，并创建任务 JSON
2. App 再调用“启动流水线”接口
3. 后端启动一个新的 Python 子进程，执行 `trainer/pipeline.py`
4. `pipeline.py` 依次调用：
   - `ffmpeg` / `ffprobe`
   - `colmap`
   - `ns-process-data`
   - `ns-train`
   - `ns-export`
   - `trainer/sam2_masking.py`
   - `trainer/object_pruning.py`
   - `npx @playcanvas/splat-transform`
5. 训练完成后，模型输出到 `storage/models/<task_id>/`
6. Viewer 通过后端静态资源和模型 URL 加载产物
7. 发布成功后，Flutter 首页和详情页会展示该商品和对应 Viewer

## 1.4 推荐给新同学的理解方式

建议把当前项目拆成四层来理解：

1. **App 层**：负责业务交互和用户流程
2. **API 层**：负责任务状态、文件路径、Viewer 配置、发布状态
3. **Pipeline 层**：负责真正的预处理、训练、导出
4. **Storage 层**：负责所有中间数据和模型结果落盘

对于继续接手电商业务的同学，最重要的是先建立以下认知：

1. 当前 MVP 的“数据真相”主要在 `storage/tasks/*.json`
2. 任务系统当前是**文件驱动**，不是数据库驱动
3. 训练链路当前是**手动启动**，不是自动队列消费
4. Object Masking 已接入，但当前实现是“训练前交互 + 训练后对象高斯提取”，不是纯粹的“先抠掉背景再训练”
5. Viewer 是后端直接静态托管，不需要单独前端构建

## 1.5 与旧文档不一致、但以当前代码为准的点

以下内容是这次交接里最容易误导新同学的地方：

1. **流水线默认不会自动启动**
   - 当前代码 `backend/app/routes/reconstructions.py` 中 `AUTO_START_PIPELINE = False`
   - 上传视频后，必须再调用一次“启动流水线”接口
   - `trainer/worker.py` 虽然存在，但当前默认流程没有使用它

2. **Object Masking 已经不是“预留能力”**
   - 旧文档 `docs/11-训练任务手动启动与迭代配置优化报告.md`、`docs/14-SAM2对象Masking接入报告.md` 中部分描述已经过时
   - 当前代码已经支持 Mask 点选、预览、确认、恢复流水线，并在导出后执行对象高斯提取

3. **旧文档里“暂不实现 3D pruning”的结论已经过时**
   - 当前 `trainer/object_pruning.py` 已用于高斯对象提取
   - 最近一次成功任务 `task_20260408_222937_2a5ff8` 的日志中已经出现 `Object-only Gaussian extraction completed`

4. **训练环境请优先使用仓库内的 requirements 文件**
   - 当前已经补充了 `trainer/requirements*.txt`，用于团队快速对齐训练与 SAM2 环境
   - `trainer/environment.yml` 仍然只是历史最小骨架，不能直接代表完整训练环境
   - 真正能跑通当前仓库，仍需按文档顺序安装 `torch / torchvision / gsplat / nerfstudio / open3d / SAM2` 等依赖

5. **当前 Flutter App 默认首页已经不是旧版 Publish Demo**
   - 当前入口页面是 `MarketplaceShellPage`
   - 旧版 `PublishPage` 仍保留在“开发者中心”中，便于调试

### 2.8.1 `trainer/pipeline.py` 对 SAM2 Python 的发现逻辑

当前代码查找顺序如下：

1. 若设置了 `SAM2_PYTHON`，优先使用
2. 否则尝试以下固定路径模式：
   - `D:\\miniconda3\\envs\\<env_name>\\python.exe`
   - `C:\\ProgramData\\miniconda3\\envs\\<env_name>\\python.exe`
   - `%USERPROFILE%\\miniconda3\\envs\\<env_name>\\python.exe`
   - `%USERPROFILE%\\anaconda3\\envs\\<env_name>\\python.exe`
3. 再退回 `conda run -n <env_name> python`

默认环境名是：

```text
sam2_app
```

### 2.8.2 `trainer/pipeline.py` 对对象提取 Python 的发现逻辑

查找顺序与 SAM2 类似：

1. 先读 `PRUNING_PYTHON`
2. 否则尝试 `3dgs_app` 的 Conda Python
3. 若当前 Python 已具备 `numpy` 和 `PIL`，则也可能直接复用当前 Python

### 2.8.3 `trainer/colmap_compat.py` 对 COLMAP 的发现逻辑

查找顺序如下：

1. 若设置了 `COLMAP_COMPAT_REAL_EXE`，优先使用
2. 否则尝试 `where colmap`
3. 最后回退到硬编码默认值：

```text
D:\colmap-x64-windows-cuda\bin\colmap.exe
```

所以如果新机器没有把 `colmap` 加入 PATH，至少要保证以下两者之一成立：

1. 手动设置 `COLMAP_COMPAT_REAL_EXE`
2. 安装路径正好兼容默认硬编码

# 4. 从零搭建环境的完整教程

## 4.1 克隆项目

```powershell
git clone <你的仓库地址> 3dgs-app
cd 3dgs-app
```

## 4.2 准备目录结构

## 4.3 安装 Git / Python / Conda

建议先安装：

1. Git
2. Python 3.13.x
3. Miniconda

安装完成后先验证：

```powershell

conda --version
```

## 4.4 创建后端虚拟环境

当前后端实际运行在仓库内的 `backend/.venv` 中。
建议在仓库根目录执行：

```powershell
py -3.13 -m venv backend\.venv
backend\.venv\Scripts\python.exe -m pip install --upgrade pip
```

然后安装后端依赖。当前仓库已经提供可直接安装的 requirements 文件：

```powershell
backend\.venv\Scripts\python.exe -m pip install -r backend\requirements.txt
```

当前 `backend/requirements.txt` 已按本机已验证版本固定。

## 4.5 创建训练主环境 `3dgs_app`

### 4.5.1 先创建基础 Conda 环境

建议先手动创建基础环境。
当前团队快速配置建议以以下文件为准：

- `trainer/requirements-cu118.txt`：训练环境的 `torch / torchvision`
- `trainer/requirements.txt`：训练环境基础 Python 依赖
- `trainer/requirements-gsplat-cu118.txt`：Windows + CUDA 11.8 下的 `gsplat` 预编译 wheel

`trainer/environment.yml` 仍保留在仓库中，但它只是历史最小骨架，不建议再作为唯一安装入口。

先创建 Conda 环境：

```powershell
conda create -n 3dgs_app python=3.10 -y
conda activate 3dgs_app
python -m pip install --upgrade pip setuptools wheel
```

### 4.5.2 安装训练核心依赖

先安装 PyTorch / CUDA 对齐版本：

```powershell
pip install -r trainer\requirements-cu118.txt
```

再安装训练基础依赖：

```powershell
pip install -r trainer\requirements.txt
```

最后强制把 `gsplat` 切到当前项目已验证的预编译 wheel，避免 Windows 上首次训练时触发本地 CUDA 编译：

```powershell
pip install --force-reinstall --no-deps --index-url https://docs.gsplat.studio/whl/pt21cu118 -r trainer\requirements-gsplat-cu118.txt
```

说明：

1. 当前项目已验证的训练组合是 `torch 2.1.2 + cu118`、`torchvision 0.16.2 + cu118`、`nerfstudio 1.1.5`、`gsplat 1.4.0+pt21cu118`

# 6. 如何跑通 3DGS 流水线

本节以当前代码真实行为为准，描述从上传视频到最终 Viewer 查看的完整闭环。

## 6.1 当前 MVP 的推荐操作入口

当前 App 里建议从以下路径进入：

1. 打开 App
2. 进入“我的”页
3. 进入卖家发布流或开发者中心
4. 创建/查看任务

如果是面向交接同学做技术验证，建议优先使用：

1. `SellerPublishFlowPage`：更贴近当前业务流
2. `TaskStatusPage`：适合观察任务状态、档位、Masking 等技术细节
3. `DeveloperHubPage`：保留了旧版技术调试入口

## 6.2 从上传视频到训练完成的完整流程

### 步骤 1：上传视频，创建任务

App 调用：

```text
POST /api/v1/reconstructions
```

后端会做的事：

1. 校验上传内容是不是视频
2. 用 `ffprobe` 读取视频元数据
3. 校验时长不超过 60 秒
4. 在 `storage/uploads/<task_id>/source.mp4` 保存原视频
5. 在 `storage/tasks/<task_id>.json` 创建任务记录

此时任务状态通常是：

```text
uploaded
```

### 步骤 2：手动启动流水线

App 调用：

```text
POST /api/v1/reconstructions/{task_id}/pipeline/start
```

可选参数包括：

1. `quality_profile`：`fast` / `balanced` / `quality` / `raw`
2. `train_max_steps`：`7000` 或 `30000`
3. `object_masking_enabled`：是否启用 Object Masking

当前代码限制：

1. `raw` 档位不能启用 Object Masking
2. 任务不会因为上传成功而自动训练，必须显式点“启动”

### 步骤 3：预处理

`trainer/pipeline.py` 会进入预处理阶段，核心包括：

1. 用 `ffmpeg` 生成归一化视频
2. 用 `ns-process-data video` 做抽帧、COLMAP、数据集整理

实际依赖命令：

```text
ffmpeg
ffprobe
colmap
ns-process-data
```

预处理完成后，常见产物位于：

```text
storage/processed/<task_id>/
storage/processed/<task_id>/dataset/
storage/processed/<task_id>/dataset/transforms.json
storage/processed/<task_id>/dataset/images/
```

任务状态通常会经历：

```text
queued -> preprocessing
```

### 步骤 4：如启用 Object Masking，则进入交互暂停

如果启动时勾选了 Object Masking，当前流水线在预处理之后会暂停，并进入：

```text
awaiting_mask_prompt
```

此时后端会从视频中选取一帧作为交互帧，前端可以：

1. 在图片上点前景/背景点
2. 请求预览
3. 确认 Mask

相关接口包括：

```text
POST /api/v1/reconstructions/{task_id}/mask-preview
POST /api/v1/reconstructions/{task_id}/mask-confirm
POST /api/v1/reconstructions/{task_id}/mask-debug
```

确认后状态会继续推进。

### 步骤 5：SAM2 掩码生成

确认 Mask 后，流水线继续执行 SAM2 全视频掩码推理。
日志里可以看到类似信息：

```text
SAM 2 masks prepared for post-training object extraction
```

这里要特别说明当前真实实现：

1. 流水线会基于用户交互生成全视频 masks
2. 但当前最终训练阶段会恢复为“全场景输入”
3. 训练结束后，再根据 masks 做“对象高斯提取”

所以它不是“先抠除背景再训练”的纯 2D Masking 训练方案，而是：

```text
交互式掩码 -> 全视频 mask 准备 -> 全场景训练 -> 导出后做对象高斯提取
```

### 步骤 6：Nerfstudio 训练

核心命令是：

```text
ns-train splatfacto ...
```

当前代码内置的档位如下：

| 档位 | 最大长边 | 训练 downscale | 模型 num-downscales |
| --- | --- | --- | --- |
| `fast` | 1600 | 2 | 2 |
| `balanced` | 2560 | 2 | 2 |
| `quality` | 3200 | 2 | 1 |
| `raw` | 不限制 | 1 | 1 |

还有一个很重要的真实行为：

1. 代码会根据输入视频分辨率做“自适应降采样”
2. 所以你明明选了 `quality`，最终实际训练 downscale 也可能变成 `1`
3. 例如最近已跑通的任务 `task_20260408_222937_2a5ff8`，最终 `effective_train_downscale_factor = 1`

训练状态通常是：

```text
training
```

### 步骤 7：导出模型

训练完成后会执行：

1. `ns-export gaussian-splat`
2. 若启用对象提取，则执行 `trainer/object_pruning.py`
3. 若允许 SOG 导出，则执行 `npx @playcanvas/splat-transform`

状态通常会进入：

```text
exporting
```

最近真实成功任务 `task_20260408_222937_2a5ff8` 的模型目录中可以看到：

```text
storage/models/task_20260408_222937_2a5ff8/model_full.ply
storage/models/task_20260408_222937_2a5ff8/model.ply
storage/models/task_20260408_222937_2a5ff8/model.sog
storage/models/task_20260408_222937_2a5ff8/splat_full.ply
storage/models/task_20260408_222937_2a5ff8/splat_object_raw.ply
storage/models/task_20260408_222937_2a5ff8/splat_object_clean.ply
storage/models/task_20260408_222937_2a5ff8/viewer.json
```

说明：

1. `model_full.ply` / `splat_full.ply` 表示全场景结果
2. `model.ply` / `splat.ply` 是最终对外默认结果
3. 若对象提取成功，默认 `model.ply` 会被替换为对象提取后的结果
4. 若 SOG 导出成功，任务会优先以 `model.sog` 作为 Viewer 模型

### 步骤 8：任务完成并可查看

成功后任务状态会进入：

```text
ready
```

若随后完成发布，状态提示可能会更新为：

```text
商品已发布，可在首页和详情页中查看 3D 展示
```

## 6.3 每一步应该看到什么现象

| 阶段 | 前端/后端现象 | 关键检查点 |
| --- | --- | --- |
| 上传成功 | App 中新增任务 | `storage/uploads/<task_id>/source.mp4` 存在 |
| `queued` | 任务已开始 | 任务 JSON 中出现 `pipeline_pid` |
| `preprocessing` | 日志持续刷新 | `storage/processed/<task_id>/dataset/` 逐步生成 |
| `awaiting_mask_prompt` | 前端出现打点界面 | 任务 JSON 中有待打点帧信息 |
| `awaiting_mask_confirmation` | 可查看 Mask 预览 | 预览图和 manifest 生成 |
| `masking` | SAM2 处理进行中 | `pipeline.log` 出现 SAM2 日志 |
| `training` | 训练耗时较长 | `pipeline.log` 出现 `ns-train` 输出 |
| `exporting` | 导出模型 | `storage/models/<task_id>/` 开始出现 PLY/SOG |
| `ready` | Viewer 可打开 | `model_url`、`viewer_url` 非空 |

## 6.5 Object Masking 如何使用

当前推荐理解方式：

1. 用户在前端选一个关键帧
2. 点前景点/背景点
3. 后端用 SAM2 生成预览
4. 用户确认后，流水线恢复
5. SAM2 对视频做 mask 传播
6. 训练仍基于全场景
7. 导出后执行对象高斯提取

因此如果新同学后续要把它更深地接入电商业务，需要明确区分两件事：

1. **交互式抠主体**：已经可用
2. **严格意义上的 masked training**：当前不是这样实现的

4. 任务能到 `ready`

# 7. 常见问题与排查

## 7.1 环境问题

### 问题 1：后端能启动，但点“开始训练”马上失败

优先检查：

1. 是否先 `conda activate 3dgs_app`
2. 当前 shell 中 `ns-train` / `ns-export` / `ns-process-data` 是否可执行
3. 是否设置了 `COLMAP_COMPAT_REAL_EXE`

推荐命令：

```powershell
where ns-process-data
where ns-train
where ns-export
where colmap
```

### 问题 2：照着 `trainer/environment.yml` 装完还是跑不起来

这是当前仓库的一个已知坑点。
原因是：

1. `trainer/environment.yml` 只覆盖了最小骨架
2. 并未完整描述实际运行环境
3. 当前应优先使用 `trainer/requirements-cu118.txt`、`trainer/requirements.txt`、`trainer/requirements-gsplat-cu118.txt`

处理方式：

1. 按本文档第 4.5 节重新用 requirements 安装训练依赖
2. 若 `gsplat` 报错，重新执行预编译 wheel 安装命令
3. 以真实 import / 命令可用性为准，不要只看 yml

## 7.2 路径问题

### 问题 3：COLMAP 找不到

当前代码会：

1. 先看 `COLMAP_COMPAT_REAL_EXE`
2. 再看 `where colmap`
3. 最后回退到 `D:\colmap-x64-windows-cuda\bin\colmap.exe`

如果你的机器路径不同，最稳妥的做法是直接设置：

```powershell
$env:COLMAP_COMPAT_REAL_EXE="D:\你的路径\colmap.exe"
```

### 问题 4：SAM2 Python 找不到

优先显式设置：

```powershell
$env:SAM2_PYTHON="D:\miniconda3\envs\sam2_app\python.exe"
```

不要完全依赖代码里的默认查找逻辑。

### 问题 5：ADB 命令找不到

当前开发机上 `adb` 文件存在，但没进 PATH。
可临时执行：

```powershell
$env:Path += ";D:\Android\Sdk\platform-tools"
```

## 7.3 CUDA / gsplat / SAM2 问题

### 问题 6：Windows 下 CUDA 扩展或 `gsplat` 报错

优先检查：

1. CUDA Toolkit 是否安装
2. `nvcc --version` 是否正常
3. Visual Studio C++ 工具链是否安装
4. PyTorch CUDA 版本是否与环境相容

当前机器可工作的基线是：

1. PyTorch CUDA 11.8
2. 本地 `nvcc` 指向 11.8
3. Visual Studio 2022 已安装

### 问题 7：SAM2 首次运行很慢或失败

可能原因：

1. 首次下载 Hugging Face 模型
2. 公司网络限制
3. `sam2_app` 环境没有配好

建议：

1. 先单独激活 `sam2_app` 做一次 import 测试
2. 如有必要，改成显式本地 checkpoint

## 7.4 COLMAP 问题

### 问题 8：COLMAP 日志里出现 BA 无法使用 GPU

当前仓库已有专门说明，见：

- `docs/12-COLMAP位姿恢复与BA无法使用GPU遗留问题报告.md`

当前结论是：

1. 这属于 COLMAP / Ceres 的常见现象
2. 不代表整个流程必然失败
3. 当前 MVP 实际已经可以在该前提下跑通完整任务

## 7.5 Flutter / Android 问题

### 问题 9：模拟器能连，真机连不上后端

常见原因：

1. 还在使用 `10.0.2.2`
2. 手机和电脑不在同一局域网
3. 电脑防火墙拦截了 8000 端口

真机应使用：

```text
http://<电脑局域网IP>:8000
```

### 问题 10：`local.properties` 导致 Flutter 构建失败

检查：

```text
mobile_app/android/local.properties
```

把 `sdk.dir` 和 `flutter.sdk` 改成自己的实际路径。

## 7.6 真机调试问题

### 问题 11：设备能充电但 `flutter devices` 看不到

检查：

1. 手机开发者模式
2. USB 调试
3. `adb devices`
4. Windows 驱动

如果走无线调试，也要先确保 ADB 可用。

## 7.7 Viewer / WebView 问题

### 问题 12：Viewer 打不开或白屏

优先检查：

1. 后端是否启动
2. `viewer_url` 是否非空
3. 模型 URL 是否可访问
4. 网络是否能访问 jsDelivr CDN

当前 `viewer/index.html` 使用了：

```text
https://cdn.jsdelivr.net/npm/playcanvas@2.17.2/+esm
```

也就是说：

1. Viewer 首次加载依赖外网 CDN
2. 离线环境或受限网络可能白屏

### 问题 13：改了 Viewer 静态文件但 App 里没生效

当前项目有一个手动维护的构建版本号：

1. `backend/app/routes/reconstructions.py` 中的 `VIEWER_BUILD_VERSION`
2. `mobile_app/lib/src/pages/viewer_page.dart` 中的 `_viewerBuild`
3. `viewer/` 静态资源中的 query string

如果改了 Viewer 逻辑，建议同步更新版本号，否则可能命中旧缓存。

## 7.8 训练分档与输入分辨率问题

### 问题 14：为什么选了高质量档，效果参数看起来没那么“高”

因为当前代码有“按输入视频自适应”的逻辑。
例如：

1. 低分辨率视频下，`quality` 可能自动把训练 downscale 调成 `1`
2. 最终生效值要看任务 JSON 中的 `effective_train_downscale_factor`

所以判断实际运行参数时，应以任务结果为准，而不是只看前端选项。

### 问题 15：视频为什么上传失败

当前后端对视频时长有限制：

1. 超过 60 秒会被拒绝
2. 不是合法视频类型也会被拒绝

## 7.9 已知限制和当前坑点

1. 当前任务系统基于本地 JSON 文件，不适合多人并发和生产环境
2. `trainer/worker.py` 存在，但当前默认不接管主流程
3. 训练环境现在应以 `trainer/requirements*.txt` 为主，`trainer/environment.yml` 不能直接当作唯一安装依据
4. Viewer 依赖外网 CDN，不适合完全离线环境
5. Viewer build version 需要手工同步，容易遗漏
6. `storage/` 会持续累积大文件，需要人工清理策略
7. 当前仓库没有接入数据库、鉴权、对象存储、消息队列
8. Android 真机联调依赖局域网、ADB、防火墙等外部条件

## 8.5 如果要继续扩展，建议从哪里开始

### 方向 1：把 MVP 的文件任务系统升级为正式业务后端

### 方向 2：把训练链路从“本机脚本编排”升级成正式服务

建议顺序：

1. 给 `trainer/pipeline.py` 做配置层抽象

3. 明确分离“后端 API 进程”和“训练 Worker 进程”

### 方向 3：提升 Viewer 和业务展示能力

建议顺序：

1. 统一 Viewer 版本管理，避免手工同步 build version
2. 解决 CDN 依赖，支持内网或离线环境
3. 在商品详情页进一步接入价格、文案、属性、购买流程

1. 后端 + Flutter 能联通
